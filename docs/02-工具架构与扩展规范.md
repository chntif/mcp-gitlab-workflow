# 工具架构与扩展规范

本文档描述当前工具实现模式，以及后续新增工具时的统一开发规范。

## 1. 架构分层

### 1.1 文件职责

- `src/server.ts`
- MCP `registerTool` 注册入口
- 参数校验、缺参报错、项目锁校验
- workflow 编排逻辑
- 统一错误包装 `withToolErrorHandling`
- `src/gitlab.ts`
- GitLab REST API 客户端封装
- query/body 组装
- HTTP 响应与错误处理
- `src/workflow.ts`
- 纯函数逻辑（分支名、标题、模板渲染、日志文本）
- `src/output-schemas.ts`
- 所有工具 outputSchema
- `data` 内字段级类型与描述
- `src/config.ts`
- 环境变量读取、兜底配置、锁配置

### 1.2 调用流

1. MCP 客户端发起工具调用  
2. `inputSchema` 做基础参数结构校验  
3. handler 做业务规则校验（缺参、条件必填、项目锁）  
4. 调用 `gitlab.ts` 或 workflow 组合函数  
5. `withToolErrorHandling` 统一封装 `ok/tool/data/error_type/message`  
6. SDK 按 `outputSchema` 校验并向客户端返回结构化结果  

## 2. 当前编写模式

### 2.1 工具组织

- 原子工具：`gitlab_*`
- 流程工具：`workflow_*`
- 全链路优先使用：
- `workflow_analyze_and_create_issue`
- `workflow_issue_to_mr_full`
- `workflow_requirement_to_delivery_full`

### 2.2 复用原则

- 新流程工具优先组合已有执行函数，不重复实现相同业务步骤
- `workflow_requirement_to_delivery_full` 通过组合“分析建 Issue + issue 到 MR 全链路”实现
- 本地分支同步统一走 `syncAndCheckoutLocalBranch`，避免分散执行 git 命令

### 2.3 参数策略

- 不在代码里写死业务项目 ID/分支
- 使用“参数优先、环境变量兜底”
- 缺少关键参数时返回明确 `input_error`
- 锁配置生效时，project_id 不匹配直接拒绝

### 2.4 输出策略

- 每个工具必须定义独立 outputSchema
- `data` 尽量字段级描述；GitLab 差异大的字段可用 `z.unknown()`
- 禁止仅用 `z.unknown()` 覆盖整个 `data`

## 3. 新增工具规范

### 3.1 开发步骤（必须按顺序）

1. 在 `src/gitlab.ts` 补充原子 API 方法  
2. 在 `src/output-schemas.ts` 增加输出 schema  
3. 在 `src/server.ts` 注册 tool，接入统一错误处理  
4. 执行 `npm run build`  
5. 更新 `README.md` 与 `docs/01-工具方法说明.md`  
6. 如涉及新配置，更新 `.env.example` 与 `docs/03-环境变量说明.md`

### 3.2 命名规范

- `gitlab_<verb>_<resource>`，示例：`gitlab_list_labels`
- `workflow_<goal>`，示例：`workflow_issue_to_mr_full`
- 禁止使用模糊名称（例如 `workflow_do_all`）

### 3.3 错误规范

- 业务缺参：`ToolInputError`
- 参数格式错误：`invalidParam(...)`
- 条件缺参：`missingParam(...)`
- 所有异常必须经 `withToolErrorHandling` 返回，不直接抛到 MCP 外层

## 4. 版本策略建议

- 删除工具属于 breaking change，建议主版本升级
- 开发阶段可快速迭代，但仍需在文档中标注“已移除工具”与替代路径
